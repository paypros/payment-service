🔹 2. Tracing de endpoints
- OpenTelemetry (OTel) es el estándar actual.
- Instrumenta tus endpoints con trazas automáticas: cada request genera un trace con spans.
- Puedes enviarlos a AWS X-Ray, Jaeger, o Tempo.
- Esto te permite ver:
- Qué endpoints tardan más.
- Dónde ocurren errores.
- Cómo fluye una petición entre microservicios (cuando tengas más).
👉 Minimalista: usa el starter de OpenTelemetry para Spring Boot, que ya intercepta requests automáticamente.

1. Levantar Jaeger con Docker
Ejecuta este comando en tu máquina:
docker run -d --name jaeger \
  -e COLLECTOR_ZIPKIN_HTTP_PORT=9411 \
  -p 5775:5775/udp \
  -p 6831:6831/udp \
  -p 6832:6832/udp \
  -p 5778:5778 \
  -p 16686:16686 \
  -p 4317:4317 \
  -p 4318:4318 \
  -p 14250:14250 \
  -p 14268:14268 \
  -p 14269:14269 \
  -p 9411:9411 \
  jaegertracing/all-in-one:latest


Esto levanta Jaeger con soporte para OTLP (puerto 4317), que es el que usa OpenTelemetry.

🔹 2. Configurar tu aplicación Spring Boot
En tu application.properties agrega:
otel.traces.exporter=otlp
otel.exporter.otlp.endpoint=http://localhost:4317
otel.resource.attributes=service.name=paypro


Con esto, cada request a tu API genera un trace y lo envía al collector local.

🔹 3. Probar trazas
- Levanta tu aplicación (mvn spring-boot:run o ./gradlew bootRun).
- Haz algunas peticiones:

curl http://localhost:8080/api/pagos
- Abre Jaeger en el navegador:
http://localhost:16686
- Selecciona el servicio paypro y verás los traces con latencia, spans y errores.

- En ECS/Fargate, en lugar de localhost:4317, apuntas al AWS Distro for OpenTelemetry (ADOT) Collector o a X-Ray.
- El flujo es el mismo, solo cambia el destino de las trazas.

¿Quién dispara ApiKeyFilter?
ApiKeyFilter extiende OncePerRequestFilter y tiene @Component — eso hace que Spring lo registre automáticamente en el filtro chain de todos los requests, sin necesidad de configuración adicional.

El flujo para GET /payments:

HTTP Request
    ↓
Servlet Filter Chain (Spring)
    ↓
ApiKeyFilter.doFilterInternal()   ← aquí se valida el header X-API-KEY
    ↓ (si es válido)
DispatcherServlet
    ↓
PaymentsController
Nota sobre SecurityConfig
El FilterRegistrationBean está comentado — eso hubiera sido una forma manual de registrar el filtro solo en /payments/*. Como está comentado, aplica a todos los endpoints (excepto /actuator que tiene su propia excepción en el filtro).

En resumen: @Component es suficiente para que Spring Boot lo inyecte automáticamente en cada request.
